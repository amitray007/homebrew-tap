# AGENTS.md — `amitray007/homebrew-tap`

This is a **private Homebrew tap** that hosts cask files for personal apps.
Each cask file is auto-generated by the corresponding source repo's release
pipeline — **AI agents should rarely edit cask files directly**.

## Repo layout

```
homebrew-tap/
├── Casks/
│   └── orpheus.rb          # current cask for amitray007/orpheus
├── README.md               # human-facing install/use docs
└── AGENTS.md               # this file
```

When new apps are added, each gets one file: `Casks/<app>.rb`.

## Where the binaries actually live

**Not on this repo.** Each app's `.dmg` lives on **that app's own source repo's
GitHub Releases**. The cask file here just points at the URL with a pinned
sha256. Example:

```ruby
url "https://github.com/amitray007/orpheus/releases/download/v#{version}/orpheus-#{version}.dmg"
sha256 "..."
```

This keeps the tap repo small, scales naturally to multiple apps, and means
"the canonical release" lives next to the source.

## How releases happen

A release for app `X` runs on `amitray007/X` (the source repo), **not here**.
That pipeline (e.g. `.github/workflows/release.yml` + `scripts/release.mjs`
on Orpheus) does end-to-end:

1. Bumps `package.json#version` (or equivalent).
2. Builds the `.dmg`.
3. Computes the sha256.
4. Creates a tagged GitHub Release on the source repo with the `.dmg` as
   an asset.
5. Renders the cask template from `scripts/X-cask.template.rb` (in the
   source repo) with the new version + sha256.
6. Writes the result to `Casks/X.rb` **here**.
7. Commits as `X: bump to <version>`.
8. Creates a **per-cask scoped tag** `X/v<version>` on this repo (see below).
9. Pushes both commit and tag.

Two ways the source-repo workflow gets triggered:

- **`workflow_dispatch`** — click "Run workflow" in the Actions tab; pick
  `bump` (patch/minor/major) or supply an explicit `version`.
- **Push to main with `[release]`** in the commit message:
  - `feat: cool thing [release patch]` → bumps patch
  - `feat: cool thing [release minor]` → bumps minor
  - `breaking: rework foo [release major]` → bumps major
  - `[release]` alone defaults to patch.

The workflow's own bump push (`chore(release): vX`) is skipped by the
trigger guard so it doesn't re-fire in a loop.

## Per-cask scoped tags

Each release adds a tag like `orpheus/v0.0.2` to this repo. Why scoped:

- **Multi-cask future-proofing:** unscoped `v0.0.2` would collide if two
  casks ship the same version.
- **Per-app git history:** `git log --tags="orpheus/*"` gives you a clean
  per-app release log; `git checkout orpheus/v0.0.1` rewinds to that
  cask state.
- **Cask commits already do the work** (`orpheus: bump to 0.0.2`), so
  the tag is just a stable git ref pointing at that commit.

## Auth

The tap is private. To install from it, set an env var Homebrew reads:

```sh
export HOMEBREW_GITHUB_API_TOKEN="$(gh auth token)"
brew tap amitray007/tap
brew install --cask orpheus
```

The CI workflow on each source repo pushes here using a fine-grained PAT
stored as a secret named **`HOMEBREW_TAP_TOKEN`** with `contents: write`
scope on `amitray007/homebrew-tap`.

## Adding a new app to the tap

Do this in the **source repo**, not here:

1. Add `scripts/<app>-cask.template.rb` mirroring `scripts/orpheus-cask.template.rb`:
   - `version "{{VERSION}}"`, `sha256 "{{SHA256}}"` placeholders
   - `url "https://github.com/amitray007/<app>/releases/download/v#{version}/<app>-#{version}.dmg"`
   - `verified: "github.com/amitray007/<app>/"`
   - `depends_on arch: :arm64`, `depends_on macos: ">= :sonoma"` (or whatever
     the app needs)
   - `app "<App>.app"`
   - `zap trash:` block for the user-data dirs the app touches
2. Add the source repo's release pipeline (`scripts/release.mjs` + the
   release GitHub Action). The Orpheus versions are good templates.
3. Make sure `HOMEBREW_TAP_TOKEN` is set as a secret on the new source repo.
4. Add a row to the table in this tap's `README.md`.
5. Trigger the first release. The tap will receive a new `Casks/<app>.rb`
   plus an `<app>/v<version>` tag, all automatically.

## Why no GitHub Releases on this repo

Binaries live on each source repo's releases. A "release" entry on this
tap would be ambiguous in a multi-cask world ("v0.0.2 of *what*?"). The
scoped tags (`<app>/v<version>`) provide all the version-tracking value a
GH release would, without the ambiguity.

## When to edit cask files by hand

Only when patching a **bad release** that can't be re-shipped from the
source repo immediately — e.g. fixing a wrong `sha256` due to a tampered
mirror. In that case, leave a short note in the commit message explaining
the patch. The very next normal release from the source repo will
overwrite your manual edit.
