# AGENTS.md — `amitray007/homebrew-tap`

This is a **public Homebrew tap** that hosts cask files and release binaries for personal apps.
Each cask file is auto-generated by the corresponding source repo's release
pipeline — **AI agents should rarely edit cask files directly**.

## Repo layout

```
homebrew-tap/
├── Casks/
│   └── orpheus.rb          # current cask for amitray007/orpheus
├── README.md               # human-facing install/use docs
└── AGENTS.md               # this file
```

When new apps are added, each gets one file: `Casks/<app>.rb`.

## Where the binaries live

**On this repo's GitHub Releases**, tagged as `<app>-v<version>` (e.g. `orpheus-v0.2.1`).
The cask URL points here so Homebrew can fetch without auth (source repos stay private).

## How releases happen

A release for app `X` runs on `amitray007/X` (the source repo), **not here**.
That pipeline (`.github/workflows/release.yml`) does end-to-end:

1. Builds the `.dmg` on `macos-14`.
2. Computes the sha256.
3. Creates a release on **this repo** tagged `X-v<version>` and uploads the `.dmg`.
4. Renders the cask template from `scripts/X-cask.template.rb` with the new version + sha256.
5. Writes the result to `Casks/X.rb` here.
6. Commits as `X: bump to <version>` and pushes.

The release workflow fires automatically when release-please pushes a `v*.*.*` tag
to the source repo, or can be triggered manually via `workflow_dispatch`.

## Auth

The CI workflow on each source repo pushes here using a fine-grained PAT
stored as a secret named **`HOMEBREW_TAP_TOKEN`** with `contents: write`
scope on `amitray007/homebrew-tap`.

## Adding a new app to the tap

Do this in the **source repo**, not here:

1. Add `scripts/<app>-cask.template.rb` mirroring `scripts/orpheus-cask.template.rb`:
   - `version "{{VERSION}}"`, `sha256 "{{SHA256}}"` placeholders
   - `url "https://github.com/amitray007/homebrew-tap/releases/download/<app>-v#{version}/<app>-#{version}.dmg"`
   - `verified: "github.com/amitray007/homebrew-tap/"`
   - `depends_on arch: :arm64`, `depends_on macos: :sonoma` (or whatever the app needs)
   - `app "<App>.app"`
   - `zap trash:` block for the user-data dirs the app touches
2. Wire up the release workflow to create a release on this tap repo and upload the DMG.
3. Make sure `HOMEBREW_TAP_TOKEN` is set as a secret on the new source repo.
4. Add a row to the table in `README.md`.
5. Trigger the first release — the tap will receive a new `Casks/<app>.rb` automatically.

## When to edit cask files by hand

Only when patching a **bad release** that can't be re-shipped from the source repo
immediately — e.g. fixing a wrong `sha256`. Leave a short note in the commit message.
The very next normal release will overwrite your manual edit.
