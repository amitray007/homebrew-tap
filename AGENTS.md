# AGENTS.md — `amitray007/homebrew-tap`

This is a **public Homebrew tap** that hosts casks, formulae, and release
binaries for personal apps. Each cask or formula is auto-generated by the
corresponding source repo's release pipeline — **AI agents should rarely edit
generated package files directly**.

## Repo layout

```
homebrew-tap/
├── Casks/
│   └── orpheus.rb          # current cask for amitray007/orpheus
├── Formula/
│   ├── silo.rb             # Node CLI formula
│   └── tracks.rb           # Tracks CLI + local viewer formula
├── README.md               # human-facing install/use docs
└── AGENTS.md               # this file
```

When new apps are added, each gets one generated `Casks/<app>.rb` or
`Formula/<app>.rb` file.

## Where the binaries live

**On this repo's GitHub Releases**, under an app-scoped tag such as
`orpheus-v0.2.1` or `tracks-cli-v0.1.0`. The generated package URL points here
so Homebrew can fetch without auth even when a source repository is private.

## How releases happen

A release for app `X` runs on `amitray007/X` (the source repo), **not here**.
That pipeline does end-to-end:

1. Builds the app or CLI artifact.
2. Computes the sha256.
3. Creates a release on **this repo** under an app-scoped tag and uploads the artifact.
4. Renders the source repository's cask or formula template with the version and checksum.
5. Writes the result to `Casks/X.rb` or `Formula/X.rb` here.
6. Commits as `X: bump to <version>` and pushes.

The exact trigger and tag shape are owned by each source repository.

## Auth

The CI workflow on each source repo pushes here using a fine-grained PAT
stored as a secret named **`HOMEBREW_TAP_TOKEN`** with `contents: write`
scope on `amitray007/homebrew-tap`.

## Adding a new package to the tap

Do this in the **source repo**, not here:

1. Add a cask or formula template in the source repository:
   - `version "{{VERSION}}"`, `sha256 "{{SHA256}}"` placeholders
   - a URL under `github.com/amitray007/homebrew-tap/releases/download/`
   - casks install an app bundle and define appropriate cleanup;
   - formulae install the CLI payload and define an offline `test do` block.
2. Wire up the release workflow to create a release on this tap repo and upload the artifact.
3. Make sure `HOMEBREW_TAP_TOKEN` is set as a secret on the new source repo.
4. Add a row to the appropriate table in `README.md`.
5. Trigger the first release — the tap will receive the generated package file automatically.

## When to edit package files by hand

Only when patching a **bad release** that can't be re-shipped from the source repo
immediately — e.g. fixing a wrong `sha256`. Leave a short note in the commit message.
The very next normal release will overwrite your manual edit.
